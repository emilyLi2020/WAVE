# Phase Narration LoRA Contest Result

This folder records the reproducible methodology and score summary for the
`lora-phase-narration` fine-tuning proof used for the contest submission.

## Claim

Fine-tuning improved held-out completion likelihood and reference similarity
versus base Gemma while preserving WAVE's structured-output, style, and safety
gates.

## Model

- Base model: `google/gemma-4-E2B-it`
- Adapter: `lora-phase-narration`
- Training method: PEFT LoRA / QLoRA via TRL `SFTTrainer`
- Runtime hardware used: NVIDIA GeForce RTX 5080
- Quantization: 4-bit NF4 QLoRA
- LoRA target: language-model projections only
  - `self_attn.q_proj`
  - `self_attn.k_proj`
  - `self_attn.v_proj`
  - `self_attn.o_proj`
  - `mlp.gate_proj`
  - `mlp.up_proj`
  - `mlp.down_proj`

Important correction: earlier experiments accidentally targeted the Gemma
vision tower's inner `.linear` modules. The final contest run uses a regex that
targets only `model.language_model.layers.<n>...` modules. Adapter weight
inspection confirmed nonzero `lora_B` updates.

## Dataset

- Source dataset: `models/datasets/clinician-seeds/lora-phase-narration-expanded.jsonl`
- Total examples: 50
- Train/test split: 80/20
- Train examples: 40
- Held-out eval examples: 10
- Split seed: 7
- Coverage:
  - 10 examples per chunk, chunks 1 through 5
  - 5 low-intensity and 5 high-intensity examples per chunk
- Dataset composition:
  - 10 original clinician seed rows from `models/datasets/clinician-seeds/lora-phase-narration-clinician.jsonl`
  - 40 deterministic synthetic draft rows generated by `models/generate_phase_narration_synthetic.py`

Synthetic rows were used only as draft training expansion. They carry provenance
notes and should be clinician-reviewed before being treated as approved clinical
training data.

## Prompt / Output Methodology

The trainer does not feed bare input JSON to Gemma. Each training row is wrapped
as chat messages:

1. `system`: WAVE role, voice, safety, no-analysis, and JSON-only instruction.
2. `user`: chunk number, chunk purpose, patient context, and task.
3. `assistant`: strict JSON object with six narration lines.

The assistant target shape is:

```json
{"lines":["line 1","line 2","line 3","line 4","line 5","line 6"]}
```

This mirrors the app's local runtime contract: prompt for structured output,
parse JSON, then validate the parsed object against the chunk-line schema.

## Exact Commands

Generate the expanded dataset:

```powershell
cd models
uv run python generate_phase_narration_synthetic.py
```

Run the final contest fine-tune and base-vs-LoRA eval:

```powershell
cd models
$env:PYTHONUTF8="1"
uv run python train_phase_narration_lora.py `
  --data "datasets/clinician-seeds/lora-phase-narration-expanded.jsonl" `
  --include-drafts `
  --epochs 0.5 `
  --learning-rate 5e-5 `
  --lora-r 8 `
  --lora-alpha 16 `
  --output-dir "runs\lora-phase-narration\contest-r8-a16-lr5e-5-0p5"
```

Inspect the adapter update:

```powershell
$env:PYTHONUTF8="1"
uv run python -c "from safetensors.torch import load_file; sd=load_file('runs/lora-phase-narration/contest-r8-a16-lr5e-5-0p5/adapter/adapter_model.safetensors'); print({'B_total_norm': sum(float(t.float().norm().item()) for k,t in sd.items() if 'lora_B' in k), 'B_zero': sum(1 for k,t in sd.items() if 'lora_B' in k and float(t.float().norm().item()) == 0.0), 'tensor_count': len(sd)})"
```

## Hyperparameters

- Epochs requested: `0.5`
- Optimizer steps completed: 3
- Effective epoch shown by trainer: `0.6`
- Batch size: `1`
- Gradient accumulation: `8`
- Learning rate: `5e-5`
- LoRA rank: `8`
- LoRA alpha: `16`
- LoRA dropout: `0.05`
- Max sequence length: `2048`
- Warmup steps: `5`
- Optimizer: `paged_adamw_8bit`

Fractional epochs are allowed by Hugging Face / TRL. With 40 train examples,
batch size 1, and gradient accumulation 8, one epoch is roughly 5 optimizer
steps. A `0.5` epoch request rounds to 3 optimizer steps.

## Evaluation Methodology

The final training script evaluates base Gemma and LoRA Gemma on the same
held-out prompts.

Metrics:

- Completion NLL: lower is better. This is the closest LLM analogue to a
  traditional ML loss like MSE.
- Completion perplexity: exponential of NLL, lower is better.
- JSON validity rate: generated output parses as JSON.
- Line count accuracy: generated output contains exactly six lines.
- Schema pass rate: output passes WAVE's chunk-line validation.
- Style pass rate: output is patient-facing, second-person narration, not a
  clinical analysis note.
- Safety pass rate: no toxic positivity, pause markers, phase/chunk
  announcements, or unsafe medication instructions.
- Medication directive pass rate: no advice to start, stop, increase, decrease,
  double, or skip medication.
- Token F1 and ROUGE-L: rough reference similarity diagnostics.
- WAVE score: composite 0-100 score used for contest reporting.

WAVE score weights:

- 25 points: completion NLL improvement versus base
- 10 points: JSON validity
- 15 points: schema pass
- 20 points: style pass
- 20 points: safety and medication directive pass
- 10 points: reference similarity

## Contest-Friendly Summary

On a 10-example held-out set, the fine-tuned LoRA improved completion NLL from
`4.7676` to `4.7097` and perplexity from `117.63` to `111.02` compared with
base Gemma. It also improved reference similarity while preserving 100% JSON
validity, schema adherence, patient-facing style, safety, and medication
directive pass rates. The composite WAVE score improved from `67.29` to
`70.44`.

## Caveats

- This is a small proof-of-concept dataset, not a production clinical model
  validation.
- The held-out set has 10 examples, so the result is useful for a contest demo
  but not statistically strong.
- Synthetic rows should be clinician-reviewed before any clinical claim.
- The adapter is a demonstration artifact; the checked-in web demo still uses
  the local Gemma runtime and fallback bank as described in `docs/models.md`.
